`v0.1.2-demo-published`는 GitHub Release가 없는 문서성/임시 tag로 유지한다. 이 문서는 새로운 tag나 Release를 만들지 않는다.

- 실제 PG 결제 연동
- 실제 카드 환불 연동

- 실제 POS API 연동

## 추천 우선순위

## 1순위: 운영 안정화 자동 검증 확대

범위:

- `smoke_test.py`에 Pro Operations read-only endpoint 추가 확대
- Admin delivery run list/detail 조회
- Admin audit log summary 조회
- Merchant weekly report notification unread count 조회

이유:

- v0.1.1 이후 가장 중요한 것은 데모 기능이 깨지지 않는지 빠르게 확인하는 것이다.
- DB 변경과 외부 연동 없이 안정화 효과가 크다.

## 2순위: 데모 UX / 문구 정리

범위:

- `/demo`를 외부 시연자용 시작점으로 정리
- README 데모 실행 순서 보강
- Mock payment / Mock delivery / Health Alert mock 경계 문구 통일

이유:

- 기능은 충분히 많으므로 이해 비용을 낮추는 것이 데모 성공률을 높인다.

## 3순위: 환경변수 / troubleshooting 문서 정리

범위:

- `docs/environment-reference.md`
- `.env.example` 계열 보강
- `docs/operations-troubleshooting-guide.md`

이유:

- 신규 환경에서 실행 실패할 때 확인 위치가 명확해진다.

## 4순위: v0.1.2 release checklist 작성

범위:

- v0.1.2 정식 릴리스 기준 정의
- tag naming과 GitHub Release 생성 조건 명시
- `v0.1.2-demo-published`와의 구분 명시

이유:

- 이미 `v0.1.2-demo-published` tag가 있으므로 다음 공식 릴리스 이름과 범위를 먼저 정리해야 한다.

## 5순위: 외부 연동 준비 설계

범위:

- PG/payment adapter skeleton 검토
- delivery provider adapter skeleton 검토
- notification channel adapter 설계
- secret storage / consent / audit policy 문서화

이유:

- 실제 연동은 아직 이르지만, 구조 설계는 다음 단계에서 필요하다.

## 추천 MVP 범위

v0.1.2 정식 릴리스의 추천 MVP 범위:

1. Pro Operations smoke coverage 추가 확대
2. Demo walkthrough와 README 실행 순서 정리
3. Environment reference / troubleshooting guide 정리
4. Admin Batch / Delivery / Health Alert 상태 help text 보강
5. v0.1.2 release checklist와 post-release verification 템플릿 작성

이 범위는 DB migration 없이 진행 가능하며, 실제 외부 API 연동도 필요하지 않다.

## DB/migration 필요 여부

추천 MVP 범위에서는 DB schema 변경이 필요하지 않다.

DB/migration 가능성이 있는 후보:

- 실제 provider credential 저장 구조
- notification consent 관리
- 외부 delivery webhook log
- 장기 audit archive

위 항목은 v0.1.2 추천 MVP 범위에서는 제외한다.

## 외부 연동 필요 여부

추천 MVP 범위에서는 외부 연동이 필요하지 않다.

실제 외부 연동은 별도 릴리스에서 아래 정책을 먼저 정한 뒤 진행한다.

- secret storage
- 개인정보/연락처/주소 저장 범위
- 수신 동의
- 실패 retry
- webhook 검증
- 운영 rollback
