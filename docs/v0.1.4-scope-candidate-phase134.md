# v0.1.4 Scope Candidate Phase 134

## 목적

Phase 134는 v0.1.3에서 추가한 Payment / Delivery / Notification / POS adapter skeleton과 External Integration Readiness 기반을 단위 테스트로 고정하고, v0.1.4 후보 범위를 정리하는 단계다.

이번 Phase에서는 기능 코드의 동작 범위를 확장하지 않고, 실제 외부 API 호출 없이 mock/noop provider의 readiness 품질을 테스트로 보강했다.

## Phase 134 반영 내용

- Payment provider service 단위 테스트 추가
- Delivery provider service 단위 테스트 추가
- Notification provider service 단위 테스트 추가
- POS provider service 단위 테스트 추가
- External Integration Readiness service 단위 테스트 추가
- 모든 provider의 `external_calls_enabled=false` 원칙 확인
- README에 v0.1.4 후보 문서 링크 추가
- Phase 133 완료 문서에 Provider adapter unit test 진행 완료 반영

Phase 135 추가 반영:

- Admin Dashboard의 External Integration Readiness 카드를 상세화
- provider별 readiness status / mode / message 표시 개선
- area별 dry-run 결과와 `external_calls_enabled=false` 표시 강화
- `MOCK_READY`, `NOT_ENABLED`, `NOT_CONFIGURED` 상태 설명 추가
- 실제 운영 연동 완료 상태가 아니라 adapter 준비 상태임을 화면에서 명확히 표시

## v0.1.4 후보 범위

### 후보 1. Adapter readiness 화면 상세화

- Admin Dashboard 카드에서 별도 상세 화면 또는 확장 패널로 발전
- provider별 readiness, mode, dry-run status, message를 더 쉽게 비교
- 실제 외부 연동 상태가 아니라 adapter 준비 상태임을 명확히 표시
- DB 변경 없이 기존 read-only API 기반으로 가능

### 후보 2. POS / CSV import UX 고도화

- 점주가 POS mock sync와 CSV import 결과를 더 쉽게 이해하도록 화면 안내 개선
- 실패 원인, import count, 재고 반영 여부를 더 명확히 표시
- 실제 POS API 호출 없이 mock/noop boundary 유지

### 후보 3. 점주 재고 이력 리포트 개선

- Inventory Ledger와 Inventory Alerts를 점주 관점에서 요약
- 재고 변경 원인, 알림 조치, Weekly Report 연결성을 강화
- 기존 데이터 기반 read-only 개선 우선

### 후보 4. Credential boundary 설계 심화

- 실제 API key/secret/token 저장 전 보안 경계 설계
- 환경변수, secret manager, audit log, 권한 분리 범위 정리
- v0.1.4에서는 문서/스켈레톤 중심으로 제한 권장
