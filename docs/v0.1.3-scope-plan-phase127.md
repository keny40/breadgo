# v0.1.3 Scope Plan Phase 127

## 목적

`v0.1.2-demo-stabilized` 이후 다음 기능 Release인 `v0.1.3`의 개발 범위를 확정한다.

v0.1.3은 실제 외부 PG / 배송 / POS / 알림 연동을 붙이는 Release가 아니라, 실제 연동을 안전하게 붙이기 위한 adapter skeleton, mock provider, read-only 상태 확인, 외부 호출 없는 smoke mode를 준비하는 Release로 정의한다.

이번 Phase에서는 기능 코드, DB schema, migration을 변경하지 않는다.

## v0.1.3 목표

- 결제, 배송, POS, 외부 알림 연동의 공통 adapter interface를 준비한다.
- 실제 외부 API 호출 없이 mock provider로 흐름을 검증한다.
- 관리자 화면에서 external integration readiness를 read-only로 확인할 수 있게 한다.
- smoke test에서 외부 호출 없이 adapter mock 상태를 검증할 수 있게 한다.
- API key, secret, token, webhook URL을 저장하거나 노출하지 않는다.

## 포함 후보

### 1. Payment provider adapter skeleton

목표:

- 실제 PG 승인 없이 결제 provider interface와 mock provider를 만든다.
- 현재 Mock payment 흐름과 충돌하지 않게 병렬 구조로 준비한다.

후보 파일:

- `backend/app/services/payment_providers/base.py`
- `backend/app/services/payment_providers/mock.py`
- `backend/app/services/payment_providers/__init__.py`

필수 메서드 후보:

- `provider_name`
- `create_payment_intent(...)`
- `confirm_payment(...)`
- `cancel_payment(...)`
- `refund_payment(...)`
- `verify_webhook(...)`

### 2. Delivery provider adapter skeleton

목표:

- 실제 배송 접수 없이 배송 provider interface와 mock provider를 만든다.
- 현재 수동 배송 상태 변경 흐름을 유지한다.

후보 파일:

- `backend/app/services/delivery_providers/base.py`
- `backend/app/services/delivery_providers/mock.py`
- `backend/app/services/delivery_providers/__init__.py`

필수 메서드 후보:

- `provider_name`
- `quote_delivery(...)`
- `create_delivery(...)`
- `cancel_delivery(...)`
- `track_delivery(...)`
- `normalize_status(...)`

### 3. POS credential boundary / adapter skeleton

목표:

- 기존 POS provider adapter 구조를 유지하면서 credential boundary를 명확히 문서/코드 구조로 분리한다.
- 실제 token 저장은 하지 않는다.
- existing Mock POS sync와 CSV import 흐름을 깨지 않는다.

포함 후보:

- provider credential config dataclass 또는 schema 후보
- secret manager 사용 원칙 문서화
- `GENERIC_POS`는 실제 호출 없이 unsupported / not configured 상태 반환

### 4. Notification provider adapter skeleton

목표:

- 실제 이메일/카카오/Push/Slack/Discord/Webhook 발송 없이 notification provider interface를 만든다.
- 기존 in-app mock delivery와 reminder 흐름은 유지한다.

후보 파일:

- `backend/app/services/notification_providers/base.py`
- `backend/app/services/notification_providers/in_app_mock.py`
- `backend/app/services/notification_providers/__init__.py`

필수 메서드 후보:

- `provider_name`
- `validate_connection()`
- `send_message(...)`
- `normalize_delivery_result(...)`
- `verify_webhook(...)`

### 5. 외부 호출 없는 adapter mock smoke mode

목표:

- smoke test가 실제 외부 API 없이 adapter가 import / instantiate / mock response를 반환하는지 검증한다.
- 실패 시 provider name과 endpoint 또는 service 이름이 명확히 나오게 한다.

후보 검증:

- Payment mock provider ready
- Delivery mock provider quote ready
- POS generic provider safely not configured
- Notification in-app mock provider ready

### 6. 관리자 화면 adapter 상태 확인 read-only 카드

목표:

- `/admin/pro/operations` 또는 별도 카드에서 external integration readiness를 보여준다.
- 실제 provider 연결 상태가 아니라 demo 기준 mock/skeleton 상태임을 명확히 표시한다.

표시 후보:

- Payment: `MOCK_ONLY`
- Delivery: `MANUAL_OR_MOCK_ONLY`
- POS: `MOCK_POS_READY`
- Notification: `IN_APP_MOCK_ONLY`
- Secret storage: `NOT_CONFIGURED`
- External calls: `DISABLED`

v0.1.3에서는 아래를 하지 않는다.

- 실제 PG 승인
- 실제 PG 환불
- 실제 배송 접수
- 실제 배송 추적 API polling
- 실제 POS API 호출
- 실제 POS token 저장
- 실제 이메일 발송
- 실제 카카오 발송
- 실제 Push 발송
- 실제 Slack / Discord / Webhook 발송
- 대량 비동기 큐
- 실시간 AI 모델
- 운영용 secret 관리 고도화 구현
- 수신 동의 관리
- 외부 provider webhook 운영 처리
- 자동 재시도 scheduler

## 추천 MVP 범위

v0.1.3 MVP는 아래 5개로 제한한다.

1. Payment provider adapter skeleton + mock provider
2. Delivery provider adapter skeleton + mock provider
3. Notification provider adapter skeleton + in-app mock provider
4. POS credential boundary 문서화 + 기존 POS adapter 안전 상태 정리
5. Admin Operations read-only external integration readiness 카드 + smoke test 보강

이 범위는 DB 변경 없이 진행하는 것을 우선한다.

## 기능별 리스크

- Payment adapter skeleton
  - 현재 Mock payment 흐름과 혼동될 수 있다.
  - 해결: UI/API 문구에 skeleton/mock 상태를 명확히 표시한다.

- Delivery adapter skeleton
  - 수동 배송 상태 변경과 provider 상태가 충돌할 수 있다.
  - 해결: v0.1.3에서는 provider 상태를 실제 주문 상태에 반영하지 않는다.

- Notification adapter skeleton
  - in-app mock delivery와 실제 external provider adapter가 섞여 보일 수 있다.
  - 해결: channel 상태를 `IN_APP_MOCK_ONLY`로 표시한다.

- POS credential boundary
  - token 저장 설계가 구현처럼 오해될 수 있다.
  - 해결: 실제 credential 저장은 제외하고 boundary/placeholder type만 둔다.

- Admin readiness card
  - readiness가 실제 연결 성공으로 오해될 수 있다.
  - 해결: `External calls disabled` 문구를 카드에 포함한다.
